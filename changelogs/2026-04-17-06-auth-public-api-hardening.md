# 2026-04-17-06 Auth and Public API Hardening

## Як було

- На публічній сторінці у проді показувався загальний банер про недоступність API без уточнення причини.
- В адмінці сценарій `Invalid password` міг маскувати інші проблеми (наприклад rate-limit або порожній пароль).
- За наявності кількох кандидатів `apiBaseUrl` адаптер не робив retry для частини 401-сценаріїв, що могло фіксувати хибний endpoint і давати некоректну помилку входу.

## Що зроблено

- У `data-adapter.js` додано діагностичний `readiness report` (`getApiReadinessReport`) з причиною/кодом/статусом останньої перевірки API.
- У `data-adapter.js` розширено retry-логіку між кандидатами API base для:
- auth 401 з кодом `AUTH_INVALID_CREDENTIALS`;
- health/readiness 401 (`/health`, `/health/db`) для безпечного переходу до наступного кандидата.
- У `data-adapter.js` змінено `login`-flow: замість повернення `false` кидаються структуровані помилки (`AUTH_INVALID_CREDENTIALS`) для точного мапінгу в UI.
- У `app.js` додано диференційовані тексти для публічного банера API-помилок:
- мережа недоступна;
- timeout;
- проблема бази даних.
- У `app.js` bootstrap тепер використовує readiness report/error-code для вибору релевантного повідомлення.
- У `admin.js` додано окремі auth-повідомлення для `AUTH_RATE_LIMITED` та `AUTH_PASSWORD_REQUIRED`, і посилено `resolveLoginErrorMessage`.
- У `admin.js` login-handler переведено на єдиний шлях через структуровані помилки замість прямого хардкоду `Invalid password`.
- У `backend/src/routes/auth.js` додано явну backend-валідацію порожнього пароля з контрактом `400 AUTH_PASSWORD_REQUIRED`; пароль нормалізується через `trim()`.

## Що покращило/виправило/додало

- Зменшено ризик хибного `Invalid password` при розсинхронізованому `apiBaseUrl` або нетипових auth-відповідях першого кандидата API.
- Публічна сторінка тепер показує більш точний причинний стан недоступності API, що прискорює інцидент-діагностику.
- Адмінський UX став точнішим: користувач окремо бачить rate-limit, вимогу ввести пароль і сервісну недоступність.
- Backend і frontend вирівняні по контракту для порожнього пароля (`AUTH_PASSWORD_REQUIRED`), що покращує консистентність помилок між шарами.
