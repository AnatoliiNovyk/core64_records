## Як було
- Після додавання треку до релізу збереження в адмінці могло падати з `Failed to save record: Internal server error`.
- Причина: `audioDataUrl` треку зберігався цілком в одному Firestore-документі `release_tracks`; для реальних MP3/WAV це могло перевищувати ліміт розміру документа Firestore.
- Помилка доходила до глобального 500 і в UI відображалась як загальна серверна помилка без контрольованої деградації.

## Що зроблено
- У `backend/src/db/repository.firestore.js` реалізовано безпечне збереження аудіо треків:
  - додано авто-стратегію `inline/chunked` для `audioDataUrl`;
  - великі аудіо payload тепер розбиваються на частини і пишуться в окрему колекцію `release_track_audio_chunks`;
  - для `create/update/replace/delete` треків додано обслуговування chunk-даних (створення/оновлення/очищення).
- Для читання треку в стрім-ендпоінті (`/release-tracks/:releaseId/:trackId/audio`) додано hydration: якщо трек chunked, `audioDataUrl` відновлюється з чанків перед віддачею аудіо.
- У `backend/src/utils/dbError.js` розширено класифікацію storage-limit помилок (включно з Firestore-повідомленнями про максимальний розмір документа), щоб уникати generic 500 там, де це ліміт сховища.

## Що покращило/виправило/додало
- Усунуто ключовий сценарій 500 при збереженні релізу після додавання великого аудіотреку.
- Бекенд став стійкішим до великих аудіо payload у Firestore без зміни API-контракту для UI.
- У випадках реального storage-limit користувач тепер отримує більш коректну категорію помилки замість загального `Internal server error`.
