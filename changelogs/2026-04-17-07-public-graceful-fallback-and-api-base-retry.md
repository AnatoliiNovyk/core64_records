# 2026-04-17-07 Public Graceful Fallback and API Base Retry

## Як було

- На публічній сторінці у режимі `requireApi=true` при невдалій readiness-перевірці відбувався ранній `return`, через що сторінка не переходила в контрольований fallback-рендер даних.
- Для частини сценаріїв `401` на `GET /public` та readiness-запитах адаптер міг залишатися на неактуальному кандидатові API base, що підвищувало ризик хибних `API unavailable` у проді.

## Що зроблено

- У `app.js` змінено bootstrap-потік публічної сторінки:
- прибрано жорсткий early-return після `isApiAvailable=false`;
- при недоступному API зберігається статус-банер з причиною;
- якщо завантаження даних не вдалося, виконується контрольований fallback-рендер `adapter.defaults` навіть за `requireApi=true`.
- У `data-adapter.js` розширено retry-правила для `401` між кандидатами API base:
- додано безпечний retry для `GET /public`;
- додано retry для safe-методів із порожнім error code (типовий кейс для несумісних endpoint-ів без стабільного коду помилки).

## Що покращило/виправило/додало

- Публічна сторінка більше не застрягає у напівпорожньому стані при API деградації: користувач отримує контент fallback + прозорий банер причини.
- Зменшено ризик хибних `API unavailable` у проді через застарілий/неправильний `apiBaseUrl` кандидат.
- Посилено API-first поведінку з контрольованою деградацією без зміни зовнішніх контрактів endpoint-ів.
