# Admin Modal Save Visibility Hardening

## Як було

- У формі модалки редагування записи могли не зберігатися для користувача "ніби без реакції" через кілька тихих `return` у submit-потоці.
- При частині помилок з API (сесія, rate limit, payload too large) користувач отримував загальне повідомлення або не отримував достатньо чіткий сигнал.
- Обробка `401` у save-потоці не була централізовано прив'язана до сценарію завершення сесії.

## Що зроблено

- У `admin.js` додано нові i18n-повідомлення (UA/EN) для сценаріїв:
  - завершена сесія (`saveRecordSessionExpired`),
  - rate limiting (`saveRecordRateLimited`),
  - надто великий payload (`saveRecordPayloadTooLarge`).
- Розширено `isUnauthorizedApiError(...)` підтримкою backend-кодів `AUTH_REQUIRED` і `AUTH_INVALID_TOKEN`.
- Посилено `resolveCrudSaveErrorMessage(...)`:
  - окремі повідомлення для unauthorized, 429/`*_RATE_LIMITED`, 413/`PAYLOAD_TOO_LARGE`,
  - збережено попередню обробку DB unavailable, 5xx та fallback із деталями.
- У submit-обробнику модалки:
  - прибрано "тихі" гілки раннього виходу для критичних невалідних станів,
  - додано `console.warn(...)` + `alert(...)` для видимої реакції,
  - `closeModal()` перенесено одразу після успішного save,
  - у `catch` додано явну обробку unauthorized через `handleUnauthorizedSessionError()`,
  - показ помилки виконано, якщо модалка видима або цільова секція все ще підключена в DOM.

## Що покращило/виправило/додало

- Збереження в модалці перестало виглядати як "нічого не відбулося" у проблемних сценаріях submit-контексту.
- Користувач отримує більш точні та практичні причини відмови збереження (сесія, ліміт запитів, завеликий payload).
- Поведінка після успішного save стала передбачуванішою завдяки ранньому закриттю модалки та стабільнішому UX-потоці.
