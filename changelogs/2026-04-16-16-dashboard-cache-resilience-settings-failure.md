# Dashboard Cache Resilience When Settings Fails

## Як було

- На Dashboard кеш оновлювався послідовними запитами: `releases`, `artists`, `events`, `sponsors`, `settings`.
- Якщо падав хоча б один запит (особливо `settings`), весь `refreshCache()` завершувався помилкою.
- Через це лічильники на Dashboard могли лишатися нулями, навіть коли основні колекції фактично були доступні.

## Що зроблено

- У `admin.js` перероблено `refreshCache()` на незалежне читання кожної колекції.
- Додано безпечний fallback на поточний кеш для окремого запиту, який не вдався.
- `AUTH`-помилки не маскуються: вони як і раніше пробиваються вище (для коректного logout/session handling).
- Додано правило: якщо не завантажилась жодна базова колекція (`releases`, `artists`, `events`, `sponsors`), кидається помилка, щоб UI показав системну проблему.
- Якщо впав лише `settings`, дані по основних колекціях продовжують відображатися.
- Оновлено cache-busting версію підключення `admin.js` у `admin.html`.

## Що покращило/виправило/додало

- Dashboard перестає «обнулятися» через одиничний збій `settings`.
- Адмінка стає стійкішою до часткових відмов API.
- Візуально зменшено сценарії, коли сторінки виглядають порожніми при наявних даних у базових колекціях.
