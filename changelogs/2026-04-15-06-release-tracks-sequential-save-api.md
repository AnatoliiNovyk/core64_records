# Release Tracks Sequential Save API

## Як було

- Збереження треків релізу виконувалось одним великим `PUT /api/release-tracks/:releaseId` із повним масивом `tracks`.
- Для кількох MP3/WAV треків сумарний JSON payload легко перевищував інфраструктурний ліміт запиту, що призводило до `Payload is too large to save`.
- Через це Save у модалці релізу міг стабільно падати навіть при валідних треках.

## Що зроблено

- Додано по-трекові backend endpoint-и:
  - `POST /api/release-tracks/:releaseId` — створення одного треку,
  - `PUT /api/release-tracks/:releaseId/:trackId` — оновлення одного треку,
  - `DELETE /api/release-tracks/:releaseId/:trackId` — видалення одного треку.
- У репозиторії додано відповідні методи для single-track CRUD.
- У `data-adapter.js` змінено `saveReleaseTracks(...)`:
  - треки зберігаються послідовно по одному запиту,
  - зайві старі треки видаляються окремими `DELETE`,
  - після операцій виконується фінальний `GET` для повернення актуального списку,
  - залишено fallback на legacy bulk `PUT /api/release-tracks/:releaseId` для сумісності зі старішим backend (лише для `404 API_ROUTE_NOT_FOUND`/`405`),
  - додано відновлення при застарілому `trackId`: якщо `PUT` повертає `RELEASE_TRACK_NOT_FOUND`, виконується `POST` для перевставки треку.
- Для release-track мутацій збільшено ефективний ліміт запитів у вікні rate-limit, щоб серійне збереження багатьох треків не блокувалось `429` у нормальному сценарії.

## Що покращило/виправило/додало

- Прибрано головний сценарій падіння save через один надто великий payload.
- Збереження релізу з багатьма треками стало стійкішим, бо кожен запит несе лише один трек.
- Забезпечено зворотну сумісність під час rolling deploy через fallback на старий bulk endpoint.
- Зменшено ризик допоміжних збоїв при серійному save (stale `trackId`, локальний rate-limit на серію мутацій).
