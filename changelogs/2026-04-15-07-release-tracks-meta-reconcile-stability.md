# Release Tracks Meta Reconcile Stability

## Як було

- Після переходу на послідовне збереження треків `saveReleaseTracks(...)` все одно робив повний `GET /api/release-tracks/:releaseId` для звірки існуючих треків і фінального результату.
- Для релізів із важкими `audioDataUrl` цей повний `GET` міг падати помилкою `500`, навіть коли окремі `POST/PUT/DELETE` вже були робочими.
- У підсумку save у модалці релізу міг завершуватись загальною помилкою `Failed to save record` через крок звірки, а не через самі мутації.

## Що зроблено

- У backend додано lightweight-метод `listReleaseTrackMetaByReleaseId(...)` без вибірки `audio_data_url`.
- Додано новий endpoint `GET /api/release-tracks/:releaseId/meta`, який повертає компактні метадані треків для звірки (`id`, `releaseId`, `title`, `durationSeconds`, `sortOrder`).
- У `data-adapter.js` оновлено `saveReleaseTracks(...)`:
  - звірка на видалення тепер використовує `GET /release-tracks/:releaseId/meta` замість важкого повного `GET`,
  - якщо `meta` недоступний на старішому backend (`404 API_ROUTE_NOT_FOUND`/`405`), використовується попередній fallback,
  - якщо сервер повертає `413` або `5xx` на етапі читання метаданих, save продовжується без кроку видалення, щоб не ламати весь сценарій,
  - фінальний результат повертається з already-resolved треків після `POST/PUT`, без повторного важкого `GET`.

## Що покращило/виправило/додало

- Прибрано критичну залежність успішного save від важкого `GET` із великими base64-аудіо полями.
- Знижено ймовірність загального падіння save у релізах із великими треками, коли мутації вже відпрацьовують коректно.
- Підвищено стійкість під час змішаного/rolling deploy завдяки сумісним fallback-сценаріям для старіших backend-версій.
