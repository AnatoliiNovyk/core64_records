## Як було
- На проді збереження EVENT блокувалося через `DB_STORAGE_LIMIT_REACHED`.
- У проєкті не було штатного інструмента для швидкої оцінки та очищення великих inline `data:*;base64,...` полів у БД.

## Що зроблено
- Додано maintenance-скрипт `backend/scripts/cleanup-inline-assets.mjs`.
- Скрипт підтримує:
  - dry-run за замовчуванням (без змін у БД);
  - `--apply` для фактичного очищення inline data URL;
  - `--include-audio` для додаткового очищення `release_tracks.audio_data_url` (окремо, як свідомо деструктивний режим).
- Додано npm-команду `db:cleanup:inline-assets` у `backend/package.json`.

## Що покращило / виправило / додало
- З'явився контрольований і відтворюваний шлях розблокувати БД при storage limit інцидентах.
- Є безпечний попередній аудит обсягу проблеми (rows/bytes) перед застосуванням змін.
- Операційне відновлення стало швидшим: можна спочатку оцінити, потім точково виконати cleanup.
