# Adapter API Base Failover Recovery

## Як було

- Після попередніх змін `data-adapter.js` міг блокувати роботу фронтенду/адмінки в сценаріях, де базовий API endpoint з конфігу був недоступний або невірний.
- У деяких деплой-сценаріях (особливо при розділених доменах frontend/backend) клієнтський код не завжди переходив на альтернативний API base автоматично.
- Через кеш браузера оновлений adapter міг не підхоплюватися відразу.

## Що зроблено

- У `data-adapter.js` перероблено логіку вибору API base:
  - зібрано упорядкований список кандидатів (`query`, `runtime`, `stored`, `config`, `/api`) з дедуплікацією;
  - додано підтримку failover між кандидатами під час `apiRequest`.
- Для `apiRequest` додано контрольовану retry-логіку на наступний API base для recoverable-сценаріїв:
  - мережеві/timeout збої;
  - `404/405` (типовий індикатор wrong-base);
  - невалідний JSON від non-API відповіді;
  - частина `5xx` сценаріїв для безпечних/auth endpoint.
- Збережено керований toggle `preferStoredApiBase` (з backward-compatible дефолтом `true` у відсутності конфіг-значення).
- Піднято cache-busting версії скриптів:
  - `index.html`: `data-adapter.js?v=2026-04-16-19`, `app.js?v=2026-04-16-19`;
  - `admin.html`: `data-adapter.js?v=2026-04-16-19`, `admin.js?v=2026-04-16-19`.

## Що покращило/виправило/додало

- Фронтенд і адмінка стали стійкішими до неправильного або застарілого API base.
- Зменшено сценарії «порожньої» сторінки через збій першого endpoint без автоматичного переключення.
- Новий adapter примусово оновлюється в браузері завдяки новому cache-busting.
