## Як було
- Після завантаження logo для header у Settings натискання Save могло падати з помилкою про ліміт сховища БД.
- Перевірка на фронтенді дозволяла великі зображення (до 2MB на файл), але реальний Firestore settings-документ має жорстко менший безпечний поріг.
- Користувач отримував помилку лише після запиту до API.

## Що зроблено
- У `admin.js` додано frontend preflight для settings save:
  - оцінка розміру майбутнього Firestore settings-документа перед відправкою на API;
  - блокування save, якщо оцінений розмір перевищує безпечний поріг.
- Додано перевірку на завеликий logo Data URL у Settings (hero/header/footer):
  - при upload завеликий Data URL відхиляється одразу на клієнті;
  - перед save додатково перевіряється, що logo-поля не перевищують ліміт.
- Для користувача використано зрозуміле повідомлення `settingsSaveTooLarge` до виконання API-запиту.

## Що покращило/виправило/додало
- Усунуто типовий сценарій падіння Save у Settings після завантаження великого header logo.
- Зменшено кількість помилок `database storage limit` у цьому user-flow.
- UX покращено: користувач отримує причину і фідбек одразу на клієнті, без зайвого мережевого round-trip.
